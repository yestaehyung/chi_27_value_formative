# 임베딩 기반 상품 검색 전환 설계

**날짜:** 2026-06-22
**상태:** 설계 (검증 진행 중)
**범위:** 추천 경로의 retrieve 단계를 BM25(FTS5 trigram) → OpenAI 임베딩으로 교체

---

## 1. 문제 (왜 바꾸나)

현재 추천 경로는 `BM25(FTS5 trigram) retrieve → 태그필터 → scoring → trade-off 랭킹`이다.
retrieve 단계가 **의미를 보지 못하고 글자 trigram 겹침만** 보기 때문에, 실데이터에서
명백히 깨진 결과를 낸다.

### 측정 (2026-06-22, nv_study.db 650개, `scripts/bench_retrieval.py`)

| 질의 | BM25 trigram 1위 | OpenAI 임베딩 상위 |
|------|------------------|--------------------|
| "운동 가성비 무선 이어폰" | 🔴 삼성 **노트북** | 🟢 골전도/런닝/자전거 이어폰 |
| "선물로 줄 브랜드 이어폰" | 🔴 **CCTV 랜덤발송** | 🟢 LG·마샬·오디오테크니카 |
| "조용한 노이즈캔슬링 이어폰" | 🟢 (명시 키워드라 양쪽 OK) | 🟢 + 소니 XM6 |
| "러닝 흘러내리지 않는 이어폰" | 🔴 무전기/게이밍 | 🟡 골전도(맞음), 일부 어린이 헤드폰 혼입 |

**결론:** trigram은 "노트북을 운동 이어폰으로" 추천하는 수준이라 사실상 고장. 단일 변수
(retrieve 방식)만 바꿔도 임베딩이 명백히 우수. → **임베딩 전환을 데이터로 정당화.**

trigram을 썼던 이유는 한국어 형태소(조사·복합어) 회피였으나, 그것은 "단어가 안 맞아도
글자는 겹친다" 수준일 뿐 의미는 전혀 못 본다("운동용 이어폰" ↔ "러닝 이어버드"는 글자
겹침 ≈ 0).

## 2. 결정 사항 (확정)

| # | 결정 | 근거 |
|---|------|------|
| D1 | retrieve를 **임베딩**으로 교체 | §1 측정 |
| D2 | **OpenAI `text-embedding-3-small`** | 2026-05 기준 OpenAI 최신·기본 임베딩, 다국어, 비용 ~$0.02/1M tok. 코드(`embeddings.py`) 이미 존재 |
| D3 | **한 변수만** 바꾼다 — 질의·상품 **둘 다 한국어**, 영어 변환 없음 | 깨끗한 인과 비교. 영어화는 별도 실험으로 분리 |
| D4 | 상품에 **LLM 생성 검색용 서술(description)** 추가 | 빈약·모호한 제목("게이밍 이어폰 N1") 보강. §1에서 임베딩이 진 케이스가 전부 제목 빈약 상품 |
| D5 | 서술은 **DeepSeek 스크립트**로 사전 1회 생성 (subagent 아님) | 재현성·일관성·연구 산출물. 650개 균일 배치엔 스크립트가 정석 |
| D6 | 상품 벡터 **디스크 캐시** (서버 시작마다 재임베딩 금지) | 시작 지연·비용 방지 |
| D7 | 카테고리 필터는 **이번 범위 밖** | 사용자 보류 결정 (나중에 고려) |

### 보류/제외
- 영어 임베딩 텍스트 (D3에 따라 별도 실험)
- 가치(TCV5) 단서 포함 서술 — 이것은 B1(가치-추천 연결)의 영역. 검색이 탄탄해진 뒤 별도로 얹는다
- ko-sroberta 등 한국어 전용 모델 — 우리 상품이 한영 혼용이라 다국어(3-small)가 더 적합

## 3. 검색용 서술 생성 (D4/D5)

`scripts/generate_product_descriptions.py` — **구현·검증 완료(5개 샘플)**.

- **입력:** title + categoryPath + tags + 가격대(band) — 가진 사실만
  - (현 데이터: description 0/650, brand 0/650 비어있음. 실질 신호 = title + categoryPath + tags 60%)
- **가드(hallucination 방지):** "주어진 정보로만. 배터리·방수·블루투스 버전 등 미제공 사양은
  지어내지 마라" — 프롬프트에 명시. 5개 검증서 위반 없음 확인
- **출력:** 한국어 1~2문장 → `seed_naver/product_descriptions.json` (id→서술, 원본 비파괴)
- **재현성:** 동일 프롬프트·모델(deepseek-v4). resume 지원(이미 된 건 건너뜀)

**검증 예 (5개):**
- "아즈라 게이밍이어폰 AIM 8…" → "발로란트·배그 등 FPS 게임에 적합한 인이어 타입, 마이크·DAC 포함 중저가형"

## 4. 아키텍처 (통합)

### 4.1 retrieve 인터페이스 교체 (D1)
호출처는 `search.py:94` **단 한 곳**:
```python
ids = search_index.retrieve(db, query, n=200, category=category)
```
→ 같은 시그니처의 임베딩 retrieve로 교체. **위 단계(태그필터·scoring·trade-off)는 무수정.**

```
[기존]  query → FTS5 MATCH bm25() → ids
[신규]  query → query_vector → cosine vs 상품벡터 캐시 → top-n ids
```

### 4.2 상품 벡터 캐시 (D6)
- 임베딩 텍스트 = `title + description(생성) + tags + categoryPath` (BM25 _doc과 정합)
- 빌드: 앱 시작 시 디스크 캐시 있으면 로드, 없으면 1회 생성 후 저장
- 저장 위치(택1, 구현 시 결정): `seed_naver/product_vectors.json` 또는 DB 컬럼/별도 테이블
- 캐시 키: product id + 임베딩 텍스트 해시(텍스트 바뀌면 무효화)

### 4.3 폴백 (graceful degradation)
- 임베딩 비활성(mock/키 없음) 또는 호출 실패 → **BM25로 폴백**
  - `MockLLMProvider`로 도는 24개 테스트가 깨지면 안 됨 → mock 경로는 기존 BM25 유지
- `embeddings.enabled()`가 이미 mock/무키 분기 제공

---

## 5. 동시성·성능 영향

- **상품 임베딩:** 사전 1회 + 디스크 캐시 → 런타임 0
- **질의 임베딩:** 매 턴 1회 OpenAI 호출 추가 (~0.3–0.6s). thinking과 무관, 작음
- "LLM-first/write-last" SQLite 불변식과 무관 (임베딩은 읽기 전용 메모리 연산)
